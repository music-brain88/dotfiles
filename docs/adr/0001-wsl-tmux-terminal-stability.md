# ADR-0001: WSL2 端末出力の二重化・破損とその根本原因

## Status

Accepted — 2026-06-27 起草の学びを 2026-07-09 に文書化。

当時は変更の物語（Obsidian/ClickUp 側で管理）の中で言及するに留まり、
ADR としては未執筆だった（本ファイル自体が後から遅れて作成されている）。

## Context

WSL2 環境で、ターミナル上の出力が二重に化ける症状が発生した。`echo` の出力が2回表示される、
`$$`（プロセスID展開）が正しく展開されない、など。

最初に疑ったのは **ターミナルエミュレータの銘柄** だった。「kitty に変えれば直るのでは」と
試したが直らず、次に wezterm に切り替えたところ症状が止まった。ここで「wezterm が直した」と
結論づけそうになったが、それは誤りだった。

実際に調査を進めると、原因は別の場所にあった。相棒（coding agent）は `init` 直下に常駐する
**tmux server** の中で動いており、その server セッションが **壊れた pty（ConPTY）を握り続けて
いた**。ターミナルエミュレータをどれに替えても、この tmux server 自体は生き残り続けるため、
握っている pty も壊れたままで、症状は再現し続ける。

つまり「ターミナルの銘柄を替えたら直った（ように見えた）」のは、たまたま同じタイミングで
tmux server を再起動する操作（ウィンドウを閉じる、WSL を再起動する等）を伴っていたためで、
銘柄そのものは症状の原因にも解決にも関与していなかった。

## Decision

対処は2つのレイヤーに分けて扱う。これらは**別の問題に対する別の対策**であり、混同しない。

### (a) 即時対処: tmux server の再生成

症状が出たその場でやるべきことは、壊れた pty を握っている tmux server を作り直すこと。

- `tmux kill-server`
- または `wsl --shutdown`（WSL 自体を再起動し、tmux server をゼロから起動し直す）

**効いたのは銘柄ではなく server 再生成だった。** これが本 ADR の核心。

### (b) 恒久対策: Windows 側ターミナルの WezTerm への移行（#389, commit edf1097）

(a) とは別のレイヤーの問題として、Windows 版 Alacritty には ConPTY の画面更新を
取りこぼす未解決バグ（[alacritty/alacritty#8057](https://github.com/alacritty/alacritty/issues/8057)）
があり、herdr の画面切り替え時に前画面の文字が残る **描画破損** が別途発生していた。

WSLg + Linux 版 Alacritty への移行（#383）も検証したが、未解決 upstream バグ3件への
回避策の積み上げとなり、体験品質（ウィンドウ管理・描画・IME）で不採用となった。

最終的に Windows 側ターミナルを **WezTerm** へ移行した（#389）。WezTerm は `conpty.dll` /
`OpenConsole.exe` を同梱し ConPTY 実装を自前で持つため、alacritty#8057 を踏まない。
「ピクセルは host」の原則はそのままに、emulator だけを最小修正で置き換えた形。

(a) の「出力の二重化・破損」は tmux server 側の pty 破損が原因、(b) の「画面切り替え時の
描画破損」は Alacritty 側の ConPTY 実装バグが原因——症状は似て見えるが原因は別物であり、
(b) の対応が (a) の症状を偶然にも間接的に緩和することはあっても、(a) の根本原因を
解決するわけではない。

## Consequences

- 同種の「出力が化ける・二重になる」症状が出たときの **一次疑い先は tmux server** とする。
  ターミナルエミュレータを疑うのは、tmux server 再生成を試して症状が再現することを
  確認した後でよい。
- **ターミナル銘柄の交換は診断を混乱させる。** 銘柄変更のタイミングでたまたま server も
  再起動されることが多く、「銘柄が直した」という誤った因果関係を記録に残しやすい。
  切り分けるときは、銘柄は固定したまま `tmux kill-server` 単体を先に試すこと。
- WezTerm 移行（#389）は ConPTY 描画破損という**別の問題**の根治であり、tmux server の
  pty 破損問題そのものへの対策ではない。両方を1つの解決策として混同して記録しない。
